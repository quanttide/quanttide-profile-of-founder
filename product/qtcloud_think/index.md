核心矛盾：暂停 vs 遗忘
你的直觉是对的：Message 全存，Note 遗忘

Q: 用户暂停后，Message 层要不要给 AI 看？
建议：摘要化呈现，而非原始对话

如果现在只改一点：
在暂停时强制生成”已确认清单“和”待探索清单“
这比完整的总结更快（解决速度问题），
结构比段落更清晰（解决透明度问题），
同时为未来恢复提供锚点（解决遗忘问题）。

你提到目前“总结过程不够透明，速度偏慢，结果不够理想”。这通常是由于缺乏中间状态的提炼。
●     引入“中间节点”：
●     不要等到最后一步才总结。
●     在每一轮澄清结束时，生成一个极简的中间摘要（例如： [澄清点] -> [结论] ）。


收集

目前已经是一个完整的最小循环。会澄清，允许用户选择退出。总结过程不够透明，速度偏慢，总结结果不够理想。是否允许用户在轮次过多时停下来，之后有时间精力再继续深入。但是这样又会可能过一阵子就忘记了。遗忘机制如何设计和如何发生，可能会有什么副作用，怎么避免。是不是要考虑把 Message 和 Note 都记录下来，遗忘主要发生在 Note 层。是想让 Note 可以过期。

这个容器得能容纳关于迭代自己的想法，不然怎么迭代别人。
